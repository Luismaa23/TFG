                          2. DESCRIPCIÓN DEL PROYECTO
================================================================================

MenuMatch es una aplicación web de recomendación de menús del día desarrollada
con Python y Streamlit. Su objetivo es ayudar a los usuarios a elegir el menú
más adecuado según sus preferencias dietéticas, restricciones alimentarias y
presupuesto disponible.

El sistema conecta dos tipos de actores principales:
  • Restaurantes: suben sus platos diarios a la plataforma.
  • Usuarios:     reciben recomendaciones personalizadas de menú.

La aplicación incluye un motor de recomendación heurístico basado en reglas y
está diseñada para evolucionar hacia un modelo de Machine Learning (ML),
utilizando los datos de evaluación recopilados durante la fase actual.

Objetivo académico:
  Demostrar la viabilidad de un sistema de recomendación de bajo coste
  computacional (heurístico) que sirva de baseline y genere datos etiquetados
  para un futuro clasificador ML.

  La aplicación sigue un patrón orientado a capas análogo al patrón MVVM:

  • View (pages/):        Páginas Streamlit, solo lógica de UI.

  • Model (utils/database.py, utils/menu_storage.py, utils/auth.py):
                          Acceso y persistencia de datos (SQLite + JSON).

  Gestión de estado:      st.session_state para datos de sesión por usuario.
  Navegación por roles:   st.navigation() con listas de st.Page() filtradas.

    elegiria_real (BOOL), fecha

                          6. MOTOR DE RECOMENDACIÓN HEURÍSTICO
================================================================================

Módulo: utils/recomendador_heuristico.py

El motor opera en dos fases:

  FASE 1 — Filtrado estricto (Hard Rules):
    Elimina platos/combos que:
      • Superan el presupuesto máximo del usuario
      • Incumplen restricciones positivas (vegetariano, sin_gluten, bajo_sal):
        el COMBO debe cumplirlas en TODOS sus platos.
      • Contienen alérgenos no deseados (sin_lactosa → con_lactosa=False, etc.):
        el COMBO falla si CUALQUIER plato lo contiene.

  FASE 2 — Puntuación (Soft Scoring):
    Fórmula simplificada (Rev. 4, por indicación del tutor):

      score = 1.0 - desviación_calórica - (precio_menú × 0.001)

      Donde: desviación_calórica = |cal_menú - cal_objetivo| / cal_objetivo

      Si cal_menú < 0.60 × cal_objetivo:
        score = score × 0.1   (penalización drástica por déficit severo)

    Motivación: El presupuesto actúa solo como filtro absoluto. Las calorías
    dominan el 95% del score. El precio actúa como desempate mínimo.

  Generación de combinaciones:
    • Normalización de texto con .lower().strip() y coincidencia por subcadena
      (ej: "postre" in categoria) para tolerar plurales y errores tipográficos.
    • Plantillas de menú:
        - Ligero:      [1 Primero/Entrante + 1 Segundo/Principal]
        - Completo:    [1 Primero + 1 Segundo + 1 Postre]
        - Tradicional: [1 Primero + 1 Segundo + 1 Postre + 1 Acompañamiento]
    • Fallback: si no hay categorías suficientes, cada plato individual es un
      menú de 1 plato.

  Funciones clave del módulo:
    _cumple_restricciones(combo, restricciones) → bool
    _calcular_score(combo, calorias_objetivo) → float
    recomendar_menu_heuristico(platos, restricciones) → dict | None
    recomendar_top_n(platos, restricciones, n=5) → list[dict]
    generar_combinaciones_menu(platos) → list[dict]

================================================================================
                          7. RECOGIDA DE DATOS PARA ML (FEEDBACK LOOP)
================================================================================

El sistema incluye un mecanismo de recopilación de datos etiquetados para
entrenar futuros modelos de ML:

  1. El usuario recibe una recomendación del motor heurístico.
  2. Se le presenta un formulario de evaluación (st.form) con 3 preguntas:
       a. Satisfacción general (escala 1-5)
       b. Relación calidad-precio (escala 1-5)
       c. ¿Elegiría realmente este menú? (Sí/No)
  3. Al enviar, se guarda un registro en la tabla SQLite evaluaciones_menus
     con el contexto completo: platos, precio, calorías, score heurístico,
     restricciones activas y las 3 respuestas del usuario.
  4. Simultáneamente, se hace backup en Google Sheets (fire-and-forget, con
     try-except para no bloquear la UI si falla la API).

  Futura integración ML:
    Los campos recommendation_type permitirán comparar en el futuro el
    rendimiento del heurístico ('heuristic') frente al modelo ML ('ml').

================================================================================
                         10. FLUJO COMPLETO DEL USUARIO (HAPPY PATH)
================================================================================

  Rol USUARIO:
    1. Accede a la app → pantalla de Login (0_Login.py)
    2. Se registra o inicia sesión (rol: usuario)
    3. Ve el dashboard de Inicio con estadísticas del sistema
    4. Va a Restricciones → configura presupuesto, calorías objetivo
       y restricciones dietéticas (vegetariano, sin_gluten, etc.)
    5. Va a Recomendaciones → pulsa "Buscar Menú Óptimo"
       → el motor heurístico devuelve el mejor combo + alternativas
    6. Evalúa el menú recomendado con el formulario (1-5 estrellas)
       → datos guardados en SQLite + backup en Google Sheets
    7. Opcionalmente: abre Carta Completa y hace un pedido libre
    8. Consulta su Historial de pedidos y puede valorarlos

  Rol RESTAURANTE:
    1. Inicia sesión (rol: restaurante)
    2. Va a Cargar Menú → sube platos vía formulario manual o CSV
       (nombre, precio, calorías, categoría, restricciones booleanas)
    3. Los platos quedan disponibles para todos los usuarios del sistema

  Rol ADMIN:
    1. Tiene acceso a todas las páginas anteriores
    2. Panel Admin: puede crear, editar y eliminar usuarios de cualquier rol
